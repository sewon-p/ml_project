Project Overview

From mobile telemetry to road-level traffic estimates.

UrbanFlow is an end-to-end traffic density estimation platform built from mobile probe telemetry, feature-engineered machine learning, spatial link matching, backend APIs, SQL-backed storage, and deployment-ready operational tooling.

Open Project Console View Link Map Open ML Pipeline

What This Project Does

System-first, not notebook-first

The system estimates road-level traffic density using only a single moving probe vehicle. Rather than depending on loop detectors or fixed roadside infrastructure, UrbanFlow turns noisy GPS, speed, and motion signals into structured traffic features, produces link-level predictions, stores the outputs, and exposes the results through APIs and interactive dashboards.

The goal was to build more than a model benchmark. This project covers dataset generation, model comparison, real-time inference, database integration, streaming interfaces, map-based review, and cloud-style deployment workflows in one coherent system.

Why This Matters

Why single-probe estimation is worth building

Most traffic monitoring systems depend on fixed infrastructure such as roadside detectors and cameras. UrbanFlow explores whether road-level traffic conditions can be inferred from a moving probe vehicle instead, using telemetry that is much easier to collect and deploy.

That makes this more than a modeling exercise. The system has to turn noisy mobile signals into stable features, serve predictions in real time, and provide enough tooling for the results to be inspected by other people, not only by the model author.

Client and Server Boundary

Where mobile processing stops and backend processing begins

Mobile Client

On-device collection

  • GPS, speed, heading, and accelerometer capture
  • Optional client-side fusion and window preparation
  • Supports a lighter `POST /ingest-features` path

Backend Server

Inference and delivery

  • `POST /ingest` for server-side fusion, buffering, and extraction
  • `POST /ingest-features` for lightweight feature payload inference
  • Prediction, link matching, persistence, streaming, and dashboards
Server-centric path: raw mobile telemetry is fused, buffered, and transformed on the backend before inference.
Client-assisted path: the mobile side prepares a feature window first, reducing server-side preprocessing work.

Modeling Results

Model comparison before deployment

The system compares tree-based models, sequence models, Gaussian processes, and traffic-flow baselines under the same workflow instead of assuming one model family would dominate. XGBoost delivered the strongest overall balance of accuracy, stability, and serving practicality for this task.

Feature Design

From traffic theory to usable features

  • Speed-density and traffic-flow relationships
  • Car-following proxies such as jerk, acceleration variance, and autocorrelation
  • Stop-go, braking, and slow-duration indicators
  • Lateral motion and temporal regularity features
  • K-means-based exploratory analysis for operating regimes

Engineering Highlights

What matters in the implementation

Spatial Optimization

Grid-indexed link matching

Road links are indexed by grid cell so repeated GPS updates do not trigger full-network scans.

Runtime Efficiency

Re-match skip threshold

Link matching is recomputed only after meaningful movement, reducing repeated GIS work during live ingest.

Signal Processing

Kalman fusion for noisy telemetry

GPS and accelerometer signals are fused into a cleaner vehicle state before feature extraction and inference.

Serving Design

Physics baseline plus ML residual

The final estimate combines a traffic-theory baseline with an ML residual correction for a more interpretable serving path.

Database Layer

Async SQL persistence

Predictions, scenarios, FCD rows, and road-link metadata are stored through async SQLAlchemy and PostgreSQL.

Operations

Containerized CI/CD workflow

The repository includes Docker, test automation, image validation, and release-driven deployment steps.

Reliability and Data Handling

Runtime behavior when dependencies are partial

  • Pydantic request schemas validate inference and ingest payloads before they reach the serving path.
  • Prediction remains available when optional services such as the database or broker layer are unavailable.
  • Map endpoints fall back to demo payloads instead of hard-failing when database queries are unavailable.
  • Hosted environments disable mutating ML-pipeline actions and keep deployment assets versioned separately from code releases.
Mobile telemetry enters the system as session-scoped records, while downstream review surfaces focus on link-level traffic state and system health rather than user-facing identity.
