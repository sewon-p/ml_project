Estimating traffic density from probe vehicles

Fixed sensors cover a fraction of the road network. UrbanFlow explores whether smartphones carried by drivers — capturing speed, acceleration, and braking — can estimate how congested a road is, without any roadside infrastructure.

This is a traffic engineering research project implemented as a full data/ML system: simulation → feature engineering → model training → real-time serving → live map. Solo end-to-end.

The Problem

Why is this hard?

Traffic density — vehicles per kilometer — is the fundamental measure of road congestion. But measuring it traditionally requires loop detectors, cameras, or radar embedded in the road, which are expensive and cover only major corridors.

Probe vehicles (taxis, ride-hails, smartphones) are everywhere, but each probe only sees its own trajectory. A car driving at 50 km/h could be in light traffic or moderate traffic — the speed alone doesn't tell you. The FD (Fundamental Diagram) speed-density relationship is degenerate in the free-flow regime: many densities map to nearly the same speed.

This project asks: can car-following behavioral patterns — acceleration variance, braking frequency, speed oscillations — break through that degeneracy?

Single probe = partial observation One trajectory is a noisy sample of the traffic stream. Systematic tests across 6 model families confirmed single-probe accuracy plateaus.
Multi-probe → +47% improvement Combining 5 probes via CF-weighted ensemble breaks through the single-probe ceiling (R² 0.46 → 0.67).
Route mismatch in real deployment Probes take different paths. Feature-level fusion requires same segment → prediction-level Bayesian ensemble as practical compromise.
Observation length matters 250m→1km improves accuracy. System chains consecutive links to reach 1km before predicting.
Approach

Car-following theory meets gradient boosting

Car-following models (IDM, Newell) tell us that a vehicle's acceleration depends on its spacing and relative speed to the leader. Higher density means more frequent interactions, which leave measurable fingerprints in the trajectory: more braking events, higher acceleration variance, speed oscillations. We engineer 31 features that capture these patterns and train XGBoost to map them to density.

31 features engineered from car-following theory: speed statistics, acceleration patterns, braking frequency, stop duration, lateral dynamics, FFT, and sample entropy. Features are registered via decorator pattern and selected through YAML config — experiments can toggle features without code changes.

Multiple probes on the same road are combined via CF-weighted Bayesian ensemble within a 15-minute rolling window. Probes exhibiting car-following behavior receive higher ensemble weight because their trajectories carry more density information.

Link-based accumulation Chain consecutive road links until 1km, then extract features and predict
Multi-probe ensemble CF-weighted Bayesian — car-following probes trusted more than free-driving
6 model families compared XGBoost, LightGBM, LSTM, CNN-1D, GPR, FD baselines under same pipeline
SHAP feature analysis Interpretable importance ranking → speed_mean, ax_std, brake_ratio dominate

Engineering Detail

How each subsystem actually works

Data Pipeline

35,000 SUMO scenarios generate 176K floating-car trajectories (6 channels × 1Hz). Each scenario parameterizes demand (200–6000 veh/h), speed limit (30–100 km/h), and lane count (1–3). Ground truth density is computed via Edie's generalized definition over the full time-space region.

Feature extraction runs through a registry pattern — each of 31 features is a decorated function selected by YAML config. This lets experiments toggle features without code changes. Training uses GroupKFold by scenario_id to prevent data leakage between probes from the same road.

All stages write to versioned run directories with manifest tracking. The ML pipeline dashboard supports resuming from any stage — retrain on existing features, re-evaluate a saved model, or filter by road conditions.

Real-Time Streaming & Ingestion

The smartphone buffers 30 seconds of GPS+accelerometer at 1Hz locally, then sends a bulk POST to reduce network calls by 30×. The server processes each sample through:

1. Server-side Kalman Filter — Raw GPS (σ=5m) + accelerometer fused into clean 2D state [x, vx, y, vy]. Heading-rotated equirectangular projection. All fusion happens server-side.
2. GIS link matching — Grid-indexed spatial lookup (<1ms) against 2.2K MOCT arterial links. Re-match only when moved >30m. Auto-detects speed_limit and lanes.
3. LinkBuffer accumulation — FCD records chain across consecutive links. Prediction triggers at 1km accumulated distance. "Sticky link" prevents GPS jitter from causing false link transitions.
4. Fan-out — Result stored to PostgreSQL, pushed via WebSocket, published to Kafka/Pub-Sub. Each channel fails independently without blocking the others.
Optimization Decisions
GIS matching Grid index (0.001° cells) reduces O(2.2K) → O(9 cells)
Re-match skip 30m threshold eliminates ~90% of GIS calls
Bulk ingest 30s batch reduces HTTP calls 30× vs per-sample
Sticky link Prevents GPS jitter from triggering false traversals
Graceful degradation DB/Kafka/GIS each optional — prediction always available
Database Schema & Ensemble Logic

Async SQLAlchemy + asyncpg with 5 ORM models. EnsembleResult tracks multi-probe aggregation per link with a 15-minute rolling window (HCM analysis period).

Ensemble lifecycle: New probe prediction → find active ensemble for that link (or create) → compute CF-weighted density → update window_end. If no new probe within 15 min → freeze. Frozen ensembles are garbage-collected after 1 hour.

CF-Weighted Bayesian Ensemble

Each probe's prediction is weighted by its car-following intensity — probes actively following a leader carry more density information than free-driving probes.

$$\text{cf}_i = \sigma_{a_x} + r_{\text{brake}} + \text{CV}_{\text{speed}}$$ $$w_i = \frac{\exp(\text{cf}_i)}{\sum_j \exp(\text{cf}_j)} \quad \text{(softmax)}$$ $$\hat{k}_{\text{ensemble}} = \sum_i w_i \cdot \hat{k}_i$$

Bayesian variant: sequential update where each probe's observation uncertainty $\sigma_{\text{obs}} \propto e^{-\text{cf}}$ — higher CF intensity → lower uncertainty → more trusted. Prior from previous link estimate or global mean. Tested R² = 0.622 vs simple mean 0.601.
