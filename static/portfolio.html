ProbeDensity · End-to-End ML System

Traffic density estimation from probe vehicles

ProbeDensity estimates road congestion using only smartphone GPS and accelerometer data collected from vehicles on the road. It turns probe trajectories into link-level traffic density predictions through a full simulation-to-serving pipeline.

The project covers simulation-based data generation, feature engineering, model comparison, real-time ingestion, spatial matching, multi-probe aggregation, storage, and live visualization.

The Problem

Why is this hard?

Core Challenge

Each probe vehicle captures only a partial view of the surrounding traffic state.

Traffic density — vehicles per kilometer — is the fundamental measure of road congestion. But measuring it traditionally requires loop detectors, cameras, or radar embedded in the road, which are expensive and cover only major corridors.

Probe vehicles (taxis, ride-hails, smartphones) are everywhere, but each probe only sees its own trajectory. A car driving at 50 km/h could be in light traffic or moderate traffic — the speed alone doesn't tell you. The FD (Fundamental Diagram) speed-density relationship is degenerate in the free-flow regime: many densities map to nearly the same speed.

This project asks: can car-following behavioral patterns — acceleration variance, braking frequency, speed oscillations — break through that degeneracy?

Observation
Single probe = partial observation One trajectory is a noisy sample of the traffic stream. Systematic tests across 6 model families confirmed single-probe accuracy plateaus.
Breakthrough
Aligned multi-probe study → +47% improvement In the 5-probe, 1 km aligned setting, multi-probe aggregation breaks through the single-probe baseline (R² 0.46 → 0.67).
Deployment
Route mismatch in real deployment Probes do not share identical 1 km start/end boundaries in deployment. That forced a second contribution: a link-level fusion system that aggregates mismatched traversals after per-probe prediction.
Constraint
Mismatched windows still share overlapping links The deployed system does not require perfectly matched 1 km cuts. It predicts each traversal first, then aggregates density through the road links those different windows overlap.
Approach

Car-following theory meets gradient boosting

Core Idea

Speed alone is ambiguous. Density shows up more clearly in interaction patterns.

Car-following models such as IDM and Newell suggest that a vehicle's acceleration depends on spacing and relative speed to the leader. As density rises, those interactions become more frequent and leave visible traces in the trajectory: more braking events, higher acceleration variance, stronger speed oscillations, and longer stop behavior.

The system converts those signals into 31 engineered features spanning speed statistics, acceleration patterns, braking, stops, lateral dynamics, FFT, and sample entropy, then trains XGBoost to map them to density. For the multi-probe case, the method then works in two stages: a CF-aware weighting rule for aligned same-slice probes, and a link-level fusion system that keeps the density-estimation pipeline usable when real probes arrive with mismatched traversal boundaries.

Engineering Detail

How each subsystem actually works

Data Pipeline & ML Dataset176K samples · 31 features · GroupKFold

From simulation to training set: 35K SUMO scenarios become 176K probe trajectories, then flow through feature extraction and leakage-safe validation.

35,000 SUMO scenarios generate 176K floating-car trajectories (6 channels × 1Hz). Each scenario parameterizes demand (200–6000 veh/h), speed limit (30–100 km/h), and lane count (1–3). Ground truth density is computed via Edie's generalized definition over the full time-space region.

Feature extraction runs through a registry pattern — each of 31 features is a decorated function selected by YAML config. This lets experiments toggle features without code changes. Training uses GroupKFold by scenario_id to prevent data leakage between probes from the same road.

All stages write to versioned run directories with manifest tracking. The ML pipeline dashboard supports resuming from any stage — retrain on existing features, re-evaluate a saved model, or filter by road conditions.

Real-Time Streaming & IngestionKalman → GIS → LinkBuffer → Fan-out

Serving path: buffered phone telemetry is fused, matched, accumulated, scored, then fanned out without making any one downstream dependency a blocker.

The smartphone buffers 30 seconds of GPS+accelerometer at 1Hz locally, then sends a bulk POST to reduce network calls by 30×. The server processes each sample through:

1. Server-side Kalman Filter — Raw GPS (σ=5m) + accelerometer fused into clean 2D state [x, vx, y, vy]. Heading-rotated equirectangular projection. All fusion happens server-side.
2. GIS link matching — Grid-indexed spatial lookup (<1ms) against 2.2K MOCT arterial links. Re-match only when moved>30m. Auto-detects speed_limit and lanes.
3. LinkBuffer accumulation — FCD records chain across consecutive links. Prediction triggers at 1km accumulated distance. "Sticky link" prevents GPS jitter from causing false link transitions.
4. Fan-out — Result stored to PostgreSQL, pushed via WebSocket, published to Kafka/Pub-Sub. Each channel fails independently without blocking the others.
Optimization Decisions
Streaming optimization decisions
GIS matching Grid index (0.001° cells) reduces O(2.2K) → O(9 cells)
Re-match skip 30m threshold eliminates ~90% of GIS calls
Bulk ingest 30s batch reduces HTTP calls 30× vs per-sample
Sticky link Prevents GPS jitter from triggering false traversals
Graceful degradation DB/Kafka/GIS each optional — prediction always available
Database Schema & Rolling Link Aggregationasync SQLAlchemy · 5 ORM models · 15-min link window

Storage model: per-probe predictions stay preserved, while each road link keeps a rolling aggregation record that opens, updates, freezes, and expires.

Async SQLAlchemy + asyncpg with 5 ORM models. EnsembleResult tracks multi-probe aggregation per link with a 15-minute rolling window (HCM analysis period).

Ensemble lifecycle: New probe prediction → find active ensemble for that link (or create) → compute weighted link density → update window_end. If no new probe within 15 min → freeze. Frozen ensembles are garbage-collected after 1 hour.

Stage 1 · Aligned Multi-Probe Aggregationsame-slice research setting · R² = 0.67

Theory layer: when probes observe the same slice, stronger car-following behavior means lower uncertainty and more influence on the fused estimate.

$$\text{cf}_i = \sigma_{a_x} + r_{\text{brake}} + \text{CV}_{\text{speed}}$$ $$w_i = \frac{\exp(\text{cf}_i)}{\sum_j \exp(\text{cf}_j)} \quad \text{(softmax)}$$ $$\hat{k}_{\text{ensemble}} = \sum_i w_i \cdot \hat{k}_i$$

Each probe's prediction is weighted by its car-following intensity — probes actively following a leader carry more density information than free-driving probes.

In the aligned experiment, this weighting defines the same-slice multi-probe result: if several probes cover the same 1 km slice, richer car-following probes should influence the combined estimate more strongly than free-driving probes.

Softmax weighting: each probe's CF score passes through a softmax $w_i = \exp(\text{cf}_i) / \sum_j \exp(\text{cf}_j)$ — higher CF intensity → higher weight → more trusted. The weighting rule is the theoretical ingredient; the next problem is making it work once probes no longer share the same start/end boundaries.

Stage 2 · Overlap-Aware Link Fusion for Mismatched Traversalsdeployed 5-probe system · R² = 0.62

Deployment layer: once probes stop sharing identical 1 km observation windows, the system predicts each traversal first, then aggregates those densities through the road links their windows overlap.

Step 1 · Deployment Problem
Different vehicles observe different 1 km windows, even on the same road.

In deployment, vehicles do not stop and start their traversal window at the same place. Their measurements are misaligned but still partially overlapping, so requiring identical segments would throw away most usable multi-probe evidence.

Unequal windows on shared links
Step 2 · Post-hoc Fusion
Predict each traversal first, then weight and aggregate on overlapping links.

The system registers each traversal's density on the road links it covers, keeps a 15-minute rolling ensemble per link, and recomputes the aggregate with a softmax over CF scores. In practice, probes that show stronger car-following behavior get higher post-hoc weight inside the shared link window.

$$\hat{k}_t = f_{\theta}(x_t)$$ $$\hat{k}_{\mathrm{link}} = \sum_{t \in T(\mathrm{link})} \alpha_t \hat{k}_t,\quad \alpha_t = \frac{\exp(\mathrm{cf}_t)}{\sum_{j \in T(\mathrm{link})}\exp(\mathrm{cf}_j)}$$
First predict each traversal t, then aggregate only the traversals whose windows overlap the same link.

Lessons Learned

What Worked, What Didn't, and What Comes Next

The strongest parts are the experiment workbench, the deployable density-estimation system, and the road-network aggregation logic that keeps the method usable outside the aligned research setting. The remaining gaps are around simulation fidelity and sensor assumptions rather than missing system pieces.

The contribution is two-stage because the research setting and the deployment setting are different.

The important result is not just that multi-probe aggregation helped. In the aligned 5-probe, 1 km experiment, aligned multi-probe aggregation reaches about R² = 0.67. The second contribution is the deployable version: an overlap-aware link-level fusion system that still delivers about R² = 0.62 in the deployed 5-probe setting once probes stop sharing the same traversal boundaries.

In the current 1-probe, 1 km setup, performance repeatedly lands around R²≈0.45.

Tested across XGBoost, LightGBM, LSTM, CNN-1D, GPR (4 kernels), window features, and density weighting, the current single-probe design repeatedly converged near that range. That is a limitation of this observation setup, not a universal ceiling for every longer window, richer sensor input, or different observation design.

The current web demo leaves a lot of device-side compute unused.

Right now the system is presented in a browser-first form, so the phone behaves mostly like a thin client. If this moves into an installed mobile app or an in-vehicle system, more of the buffering, sensor fusion, feature preparation, and filtering can run locally before upload. That would reduce server load, network cost, and end-to-end latency without changing the core model idea.

1 km accumulation was a deployment choice for link-level measurement, not just an accuracy trick.

Links are short and variable, so a fixed time window does not map cleanly onto a stable road segment. The system therefore chains consecutive links until a traversal reaches about 1 km, then predicts at that road-linked unit. Unequal traversal boundaries are a consequence of deployable link measurement, not an arbitrary preference for longer windows.

If inter-vehicle gap becomes observable, the model should get much stronger.

In the aligned multi-probe study, XGBoost (31 features + CF) rises from 0.641 → 0.752 → 0.801 → 0.848. The point is not just that more probes were added, but that better-aligned vehicles reveal spacing structure. If connected vehicles or direct communication make headway or gap measurable as an input, the model would no longer have to infer that information only from trajectory shape.
